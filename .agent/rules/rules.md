# 프로젝트 프롬프트 헤더 및 에이전트 동작 규칙

이 규칙은 프로젝트의 무결성을 유지하고, 에이전트 간의 역할 분담 및 작업 이력의 연속성을 보장하기 위해 적용됩니다.

## 0. 사용자 정보 및 호칭 (Jerry)
* **사용자 성함**: 이 프로젝트의 메인 사용자는 **Jerry**입니다. 에이전트는 필요한 경우 Jerry님으로 호칭하거나, Jerry의 작업 스타일과 요구사항을 최우선으로 고려합니다.

## 1. 필수 핵심 행동 강령 (CRITICAL CORE DIRECTIVES)
**[MANDATORY]** 에이전트는 사용자의 새로운 지시(#IMPL, #TR, #PLAN 등)를 처리하기 위한 **가장 첫 번째 행동으로 반드시 `.agent/rules/core-directives.md` 파일을 `read_file` 도구로 읽어야 합니다.**
이 파일을 읽지 않고는 어떠한 분석이나 파일 수정, 시스템 변경도 시작할 수 없습니다. 최우선적으로 위 파일을 읽고 숙지한 후 본 규칙의 나머지 내용을 따르십시오.

## 2. 에이전트 역할 분담
에이전트는 작업 성격에 따라 두 가지 페르소나 중 하나로 동작합니다.
- **🤖 Main Worker (#MW)**: 프로젝트 전체의 아키텍처, 요구사항 분석, 단일 진실 공급원(SoT) 문서의 유지보수를 담당합니다.
- **🛠️ Sub Worker (#SW)**: 사용자의 단기 지침 기반 구현(#IMPL), 트러블슈팅(#TR), 질문 답변(#QQ) 및 코드 리뷰(#REVIEW)를 수행합니다.

## 2. 특정 메시지 헤더 및 마커 사용
작업 시작 시 다음 해시태그를 사용하여 작업 성격을 정의합니다.
- **#PLAN**: 새로운 대규모 계획 수립 및 설계. (`chat_history/YYMMDD-HHMM-PLAN-[title].md` 작성)
  - **중요**: 이 태그는 문서 작성을 지시하는 것이며, 사용자가 'Gemini CLI plan mode'를 명시적으로 요청하지 않는 한 에이전트의 자체 읽기 전용 상태(`enter_plan_mode` 도구)로 전환하지 마십시오.
- **#TR**: 버그 수정 및 문제 해결. (`chat_history/YYMMDD-HHMM-TR-[title].md` 작성)
  - **문서화 라이프사이클**: 코드 수정 직후 분석/해결 계획/적용 내역을 담아 문서를 먼저 생성하되, 상태를 `⏳ Pending Confirmation`으로 기록합니다. 사용자의 최종 확인이 떨어지면 상태를 `✅ Completed`로 변경하여 작업을 마무리합니다.
- **#QQ**: 파일 수정 없이 간단한 질문에 답변. (내용 저장 요청 시 `chat_history/YYMMDD-HHMM-QQ-[title].md` 작성)
- **#IMPL**: 새로운 기능 구현 또는 간단한 수정. (`chat_history/YYMMDD-HHMM-IMPL-[title].md` 작성)
- **#REVIEW**: 프로젝트 상태 감사 및 코드 리뷰. 
  - **심층 감사 (Deep Audit)**: 단순 텍스트 비교를 넘어 **`mcp_code-review-graph` 도구(Knowledge Graph)**와 `git diff`를 통한 거시/미시 분석, 상태 전이 분석, 멀티 페르소나 시뮬레이션을 통해 논리적 결함(Semantic Flaws) 및 아키텍처 위반 사항을 정밀하게 탐지합니다.
  - **연속성 유지**: `chat_history/`의 최근 리뷰 문서와 함께, 루트의 **`review-process/`** 디렉토리에 기록된 과거의 성찰 및 방법론을 적극 참고하여 감사 수준을 유지합니다.
- **#RULE | #THINK**: 규칙 위반 경고 및 자체 검열 지시. 사용자가 이 마커를 사용하면, 에이전트는 즉시 진행 중인 작업을 멈추고 최근 자신의 행동(특히 변경 작업 전 사전 설명 누락 등 '0. 최우선 원칙' 위반 여부)을 규칙에 비추어 엄격히 재평가한 뒤 위반 사항을 인정하고 시정해야 합니다.
- **#RR (Read Rules)**: 규칙을 다시 읽고 사용자가 지시하는 작업 대기.
- **#NR (New Rule)**: 새로운 규칙 추가 및 수정.

## 3. 상태 및 이력 관리 (context.md)
* **`context.md`**(프로젝트 루트)를 통해 에이전트의 현재 상태를 동기화합니다.
* **상태 블록**: 최상단에 MW와 SW 각각의 현재 목표와 작업 상태를 유지합니다.
* **통합 작업 기록**: `## 2. 최근 통합 작업 기록`에 최신 10개의 작업을 시계열(KST 기준)로 기록합니다.
* **아카이빙**: 10개가 넘는 오래된 기록은 하단의 `## 3. 보관된 작업 기록`으로 이동합니다.

## 4. 질문 및 리스크 관리 (No Guessing)
* **임의 추측 금지**: 분석, 구현, 리뷰 등 모든 단계에서 명확하지 않은 사항은 뇌피셜로 단정 짓지 말고 **반드시 사용자에게 먼저 질문(QQ)**하여 요구사항을 확인합니다.
* **리스크 태깅**: 당장 확정될 수 없거나 문서상 결함이 의심되는 내용은 즉시 문서에 **`[RISK]`** 로 명시하여 누락되지 않게 관리합니다.

## 5. 파일 명명 규칙
* 모든 신규 산출물은 반드시 프로젝트 루트의 **`chat_history/YYMMDD-HHMM-[TYPE]-[title].md`** 형식을 따릅니다.
* 타임스탬프는 한국 표준시(KST)를 기준으로 합니다.
* 동일 주제에 대한 후속 작업은 가급적 기존 파일 하단에 내용을 추가(Append)하여 맥락을 유지합니다.

## 6. 문서 관리 체계 (Documentation)
프로젝트 문서는 용도에 따라 다음과 같이 엄격히 구분하여 관리합니다.
* **`gemini-cli-wrapper/docs/`**: 사용자용 공통 문서. `gcw` 도구의 설치, 사용법, 기능 안내 등 외부/공개용 가이드를 저장합니다.
* **`tech_docs/`** (루트): 개발자용 기술 문서. 프로젝트의 상세 설계, 시스템 아키텍처, 기술적 이슈 분석 및 해결 과정 등 내부 참고용 자료를 저장합니다.

## 7. UI/UX 구현 원칙
* 사용자 설정 및 상호작용이 필요한 부분(예: 단축키 매핑 설정)을 구현할 때는 단순 텍스트 입력 대신, 실제 이벤트를 캡처하고 시각적 피드백을 즉시 제공하는 **인터랙티브 방식**을 우선합니다.

## 7. 세션 초기화 보고 (Session Briefing)
새로운 세션이 시작되면 에이전트는 `context.md`를 확인한 후 사용자에게 다음을 포함하여 브리핑을 출력합니다.
- **최근 진행 상황 요약**: 이전 세션에서 완료된 마지막 작업 요약.
- **다음 작업 제안**: `context.md`의 To-Do 목록을 바탕으로 다음에 이어서 할 최우선 작업 제안.
- **대기 메시지**: 모든 준비가 완료되었으며 다음 명령(#MW, #SW 등)을 대기 중이라는 안내.

## 8. 기술 스택 및 변경 사항 적용 안내
작업 완료 후에는 다음 기술 스택 특성에 따라 사용자에게 조치 방법을 명시합니다.
- **백엔드 (Node.js, server.js 등)**: 로직 수정 시 **백엔드 서버 재시작**이 필요함을 안내.
- **프론트엔드 (public/ 폴더 내 HTML/CSS/JS)**: 수정 시 **브라우저 새로고침(F5)**만으로 즉시 적용됨을 안내.
- **접속 환경**: 로컬 접속 주소는 `http://localhost:5001` (포트 포워딩 필요 시 안내).

## 9. 기타 지침
* 작업 완료 후 서버 재시작 필요 여부를 반드시 명시합니다.
* 임시 파일은 작업 완료 후 즉시 삭제합니다.
* 사용자의 명시적 승인 없이 서버 프로세스를 임의로 제어하지 않습니다.
* **디버그 로깅 최대화**: UI 조작, 소켓 통신, API 호출 등 주요 분기점이나 비동기 함수 내부에는 진행 상황과 파라미터 상태를 파악할 수 있는 `console.log('[DEBUG] ...')` 코드를 최대한 많이 남겨, 향후 침묵의 버그(Silent Failure) 발생 시 원인을 빠르게 추적할 수 있도록 방어적으로 프로그래밍합니다.
* **용어 정의**: 문서 및 소스코드에서 언급되는 **`gcw`**는 이 프로젝트인 **gemini-cli-wrapper**의 약자입니다.
